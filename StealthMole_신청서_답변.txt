질문 1: 이번 해커톤에서 OSINT data를 활용해 만들고 싶은 것은 무엇인가요?
--------------------------------------------------

AirMaven: 다층 OSINT 융합 기반 항공·해양 ISR 상황인식 플랫폼

공개 항공기 추적(OpenSky), 날씨(Open-Meteo), 뉴스(GDELT), 위성 궤도(CelesTrak), 해양 AIS를 융합한 실시간 상황판에 사이버 위협 인텔리전스 레이어를 추가하려 합니다.

구체적 활용:
1. 항공·해양 보안 이벤트 조기 경보
   - 다크웹 해킹 포럼의 항공사/공항 관련 위협 정보
   - 데이터 유출 사례와 실제 항공기 이상 경로 상관관계 분석

2. Fusion Copilot 강화
   - 기존: "GDELT 뉴스 + 항공기 궤적 이상" 탐지
   - 추가: "+ StealthMole 다크웹 위협 인텔" 삼각 검증

3. 타임라인 기반 위협 추적
   - 공개 OSINT(뉴스/SNS) → 다크웹 포럼 → 실제 사이버 공격 연결고리 시각화

현재는 "표층 웹" OSINT만 다루지만, StealthMole로 심층 웹 위협 신호까지 포착하여 분석가 의사결정 지원의 정확도를 높이고자 합니다.

질문 2: OSINT Track 준비 또는 활용 하면서 가장 어렵게 느껴지는 점은 무엇인가요?
--------------------------------------------------

다층 소스 융합 시 데이터 품질 불균형과 상관관계 검증

1. 기술적 도전
   - 공개 API 8개 소스(항공/날씨/뉴스/위성/해양/공항)의 rate limit, 신선도, 누락 처리가 각각 다름
   - 예: OpenSky 익명 모드 429 에러 → 캐시 재사용, GDELT rate limit → Google News RSS 대체
   - 소스마다 "사용 불가" 상태 처리와 신뢰도 표기 필요

2. 상관관계 분석의 신뢰성
   - "뉴스에서 대만해협 긴장 + 항공기 경로 이탈" 탐지 시, 우연의 일치인지 실제 연관인지 판단 어려움
   - 현재는 룰 기반 anomaly engine이지만, 다크웹/포럼까지 포함하면 False Positive 관리가 더 복잡해질 것으로 예상

3. 실시간성 vs 정확성 트레이드오프
   - 30초 폴링 주기로 "near-real-time" 구현했지만, 심층 위협 인텔은 수집 지연이 클 수 있음
   - 캐시 TTL 전략과 "오래된 정보" 표시 기준 설정 고민

StealthMole API가 구조화된 위협 인텔(severity/confidence/timestamp)을 제공한다면, 기존 공개 OSINT와의 삼각 검증으로 신뢰도를 높일 수 있을 것으로 기대합니다.
